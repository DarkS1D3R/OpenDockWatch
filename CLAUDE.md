# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

OpenDockWatch is a small self-hosted Docker monitoring dashboard: container list/start/stop/restart, live log tailing, a topology ("Flow") graph, and an alerts/events feed. It watches the local Docker daemon and any number of remote hosts over SSH. See `README.md` for the full feature list and `CONTRIBUTING.md` for the release process.

## Commands

```
npm install                                 # deps
cp .env.example .env                        # then fill in AUTH_USER/AUTH_PASS_HASH/SESSION_SECRET
cp config/hosts.example.json config/hosts.json
npm run hash-password -- "your-password"    # generates a bcrypt hash for .env's AUTH_PASS_HASH

npm run dev                                  # node --watch server/index.js, http://localhost:3000
npm start                                    # plain node server/index.js

npm test                                     # syntax-check every server/public-js/scripts file, then node --test
node --test test/docker.test.js              # run a single test file
node --test --test-name-pattern="parseByteString" test/docker.test.js   # run one test by name

npm run lint                                 # eslint .
npm run format                               # prettier --write .
npm run format:check                         # prettier --check .  (what CI runs)

docker compose up -d --build                 # run OpenDockWatch itself in a container
```

CI (`.github/workflows/ci.yml`) runs `npm run lint && npm run format:check && npm test`, then a plain `docker build`. Always run the same three before considering a change done.

There is no build step for the frontend — `public/js/*.js` is served as-is via native ES modules (`<script type="module">` in `public/index.html`). Most of it (`app.js`, `public/js/components/**`) is only syntax-checked, not unit-tested, since it's Vue/Cytoscape UI code with no meaningful behavior outside a browser - but pure logic with no DOM/cytoscape dependency has no such excuse: `format.js` (ANSI parsing, level detection, byte/rate formatting), `graph.js`'s `buildElements`/`aggregateGroups` (graph-building), and `public/js/lib/spark.js`/`lib/logStream.js` (sparkline coordinate math and the EventSource/buffering factory shared by `LogViewer`/`ContainerDetail`'s log streams - `EventSourceImpl` and the flush `schedule` are injectable specifically so this is testable under `node --test` with a stub source, without a browser) are all unit-tested. `test/format.test.js`, `test/graph.test.js`, `test/spark.test.js`, and `test/logStream.test.js` `import()` their module directly (dynamic import, since `require()` can't load an ES module - `node:test`'s `before()` hook loads each once and shares it across that file's tests). When pulling more logic out of a component in `public/js/components/`, prefer landing it in `public/js/lib/` if it's pure - that's the difference between it being tested and not.

## Architecture

**The server shells out to the `docker` CLI** (`server/docker.js`, via `child_process.execFile`/`spawn`) rather than talking to the Engine API or a Docker SDK. Every function takes a `host` object and prefixes its args with `-H <dockerHost>` when set (`hostArgs()`); a `null`/absent `dockerHost` means the local socket. This is _why_ remote hosts work at all: `-H ssh://user@host` is resolved by the Docker CLI itself using normal SSH key/config, so there's no separate tunneling code to maintain. Hosts are defined in `config/hosts.json` (gitignored, copy from `hosts.example.json`), each with an optional `edges: [{from,to,label}]` list for relationships Docker itself can't see. Hosts can also be added/edited/removed live from the Settings panel (admin-only, `/api/settings/hosts`) — `server/hosts.js`'s `saveHosts()` rewrites the file and the route calls `metricsCollector.addHost/removeHost` and `eventWatcher.addHost/removeHost` for just that one host, so a new SSH host starts being monitored immediately with no process restart (edits are a `removeHost` + `addHost` pair rather than an in-place update, since reconnecting is simpler than diffing what changed).

**`server/logger.js`** writes structured single-line audit/debug events (`[opendockwatch] [LEVEL] event key=value ...`) to stdout/stderr - picked up by `docker logs` automatically and by the app's own Log Viewer level filters. Used for auth success/failure, container start/stop/restart, settings changes, and every alert firing (`alerts.js`'s single `fire()` function). Never log secrets (webhook URLs are logged as scheme-only, e.g. `discord://…`, since they embed tokens).

**Pure parsing/graph-building functions are exported from `server/docker.js`** (`parseByteString`, `parseLabels`, `parseHealth`, `networkEdges`, `dependsOnEdges`, …) specifically so `test/*.test.js` can unit-test them directly without mocking `child_process`. When adding docker-CLI-output parsing, keep the parsing pure and separate from the function that actually shells out, and export it.

**Two long-running background jobs, started in `server/index.js` on boot:**

- `eventWatcher.js` — one persistent `docker events` process per host (auto-restarts with exponential backoff on exit/error, capped at 30s). Each parsed event line fans out to three places: `db.insertEvent` (persistence), `broadcaster.publish` (live SSE push to the Activity tab), and `alerts.handleEvent` (rule engine). This is the single ingestion point for all event-driven alerts (`container_crashed`, `crash_loop`, `unhealthy`).
- `metricsCollector.js` — polls every host every `POLL_MS` (5s) for containers/stats/info, plus disk usage every 60s; keeps an in-memory snapshot per host (`getSnapshot(hostId)`, used e.g. to avoid an extra docker call just for a container's name in audit logging) and writes samples to sqlite for history charts. Also feeds the threshold-based alerts (`container_cpu`/`container_mem`/`host_cpu`/`host_mem`/`docker_disk`, `handleSample`/`handleHostSample`/`handleDiskUsage` in `alerts.js`).

**`alerts.js`** is the rule engine plus webhook delivery. Every alert setting (webhook URL/format, CPU/mem/disk thresholds) follows the same pattern: an `.env` default, overridable from the DB via the admin Settings UI (`PUT /api/settings/...`), with the DB value always winning when present — even an explicit empty/0 to deliberately disable something `.env` configured. See the README's Alerts section for the full rule table and webhook URL-scheme routing.

**`server/db.js`** is a single `better-sqlite3` database (`data/opendockwatch.db`) holding metrics history, events, audit log, alerts, settings overrides, and doubles as the express-session store. No migration framework — schema is a set of `CREATE TABLE IF NOT EXISTS` statements run at startup; add new tables the same way. A new _column_ on an existing table needs both: add it to the `CREATE TABLE` (for a fresh database) and a `try { db.exec('ALTER TABLE ... ADD COLUMN ...') } catch {}` right after the schema block (for an existing one upgrading — `IF NOT EXISTS` only skips the whole statement, it doesn't retroactively add columns to a table that already exists without them).

**Auth (`server/auth.js`)** is cookie-session based with two roles: admin (full control) and an optional read-only viewer (`VIEWER_USER`/`VIEWER_PASS_HASH`). Role enforcement is server-side (`requireAdmin` middleware on every mutating route) — the viewer role hides buttons in the UI too, but a request forged around the UI still gets a 403 from the server.

**Frontend is Vue 3 (Options API)**, no SFCs/build step — `public/js/**` is loaded as native ES modules via `<script type="module">`. `app.js` is the root component (~400 lines): session/role bootstrap, host list/selection, the 5s polling loop, containers/stats/alerts state, view switching, and `doAction`. Everything view-specific lives in `public/js/components/`: `SparkTile`/`HostCard` (the host-usage tiles), `ContainerList`/`ContainerDetail` (the List view and its detail panel), `LogViewer` (the full-size log panel), `ActivityView` (the alerts/events tab), `SettingsPanel` (webhook/threshold/host config), and `FlowView` (the Cytoscape graph/tree view). Panels that are meaningfully transient (`LogViewer`, `ContainerDetail`, `ActivityView`, `SettingsPanel`) mount via `v-if`, each time fresh, so their own `mounted()`/`beforeUnmount()` own their stream or fetch lifecycle instead of the root orchestrating it (e.g. `ContainerDetail` watches its own `container.id` prop to reset/refetch on a same-mount container switch, since watching the whole `container` object would refire on every 5s poll — the root replaces its `containers` array wholesale each poll, so the object reference changes even when the id doesn't). `ContainerList` and `FlowView` instead stay mounted via `v-show` across List/Flow tab switches, since there's nothing external to tear down and rebuilding a Cytoscape graph from scratch on every switch would be wasteful for a host with many compose projects — they watch their own props (`topology`, `hostId`, `stateFilter`) to re-render instead of remounting. State that's genuinely shared across views (`selectedContainerId`, `selectedHostId`, `alerts`) stays in the root and flows down as props, with children emitting events back up (`select`, `action`, `close`, `open-log-viewer`, ...) rather than reaching into siblings; `fullscreen` on both `LogViewer` and `FlowView` is the one exception worth calling out — it's a real `v-model` (not fully internal) because the root's own layout needs to know about it directly, to hide the host card and other panels. `public/js/lib/` holds pure, unit-tested logic shared across components: `spark.js` (sparkline coordinate math, used by `SparkTile`) and `logStream.js` (the EventSource/buffering factory both `LogViewer` and `ContainerDetail`'s log preview build on). Supporting modules: `api.js` (the only place `fetch` is called, wraps 401→redirect-to-login and JSON/error handling), `format.js` (pure display formatting: ANSI-to-HTML for colored logs, byte/timestamp formatting, log-level detection, badge/icon lookups), `constants.js` (poll intervals, history lengths), `graph.js` (all Cytoscape.js flow-view logic: layout via dagre, edge/node styling, compose-group collapse/expand via cytoscape-expand-collapse, and `localStorage`-backed persistence of dragged node positions + camera zoom/pan + collapsed group ids, keyed per host so a reload doesn't reset your arrangement — owned by `FlowView`, which is the only component that imports it). The Flow view has two render modes sharing the same topology data: graph mode (`buildElements`, compose-group boxes, network/depends-on/manual edges) and tree mode (`buildTreeElements`, an ArgoCD-style `project → container → network/mount` DAG with no compound groups, network/volume pills deduped globally so shared infrastructure gets one node with multiple incoming edges). A tapped tree-mode pill (`proj:`/`net:`/`mount:` id) is filtered out inside `FlowView` before it ever reaches the root — routing it into `selectedContainerId` would trip `ContainerDetail`'s watcher into fetching `docker inspect` for a fake id — so it's kept as `FlowView`-local state (`pillSelection`), used only for fading and the tap-info bar. Container nodes are shared between both modes via `containerNodeEl`. Positions/camera are keyed per host _and_ per mode (`odw:flow:positions:<mode>:<hostId>`, etc.) since a dagre layout from one mode is meaningless in the other; the mode choice itself is also saved per host. Vendor libraries (Vue, Cytoscape, dagre, html2canvas-pro, cytoscape-expand-collapse) are checked into `public/vendor/` and loaded as global `<script>` tags, not npm/bundled — `eslint.config.js` declares them as browser globals for that reason. html2canvas-pro exists specifically because `cy.png()` only rasterizes cytoscape's own `<canvas>`, not the `cytoscape-node-html-label` DOM overlay that actually renders node content (name, icons, CPU/RAM bars, badges) — Flow view's "Export PNG" screenshots the real on-screen DOM instead (this means it has a fixed pixel resolution to manage - see `EXPORT_SCALE` and the container-resize dance in `exportPng`). "Export SVG" sidesteps that entirely: `extractSvgGeometry` (impure, reads the live `cy` instance's positions/classes/data) feeds `renderSvg` (pure, geometry → an `<svg>` string, unit-tested the same way `buildElements` is) which hand-draws the same node/edge visual design `CY_STYLE` and the HTML templates already encode - no dependency on the DOM overlay or a rasterization step, so there's no resolution ceiling to manage for a host with a lot of compose projects. expand-collapse physically removes a collapsed group's children from the graph, so `updateGraph`'s poll-driven diff skips them entirely while hidden rather than touching them (an earlier version briefly expanded/re-collapsed every poll to keep their data fresh, but that visibly flickered the collapsed box every 5s) — `expandcollapse.afterexpand` re-syncs a group's children from the latest poll's data the moment it's actually opened instead. Below `COMPACT_ZOOM_THRESHOLD` (graph.js), nodes render a compact HTML label (state/icon/name only, no metric bars) and shrink to match via a per-element style function on node height keyed off a `compact` data flag - flipped synchronously on every `viewport` event (immediate, unlike the debounced viewport-position save alongside it) so it reads as a direct consequence of zooming.

**Live updates use Server-Sent Events**, not WebSockets: container logs (`GET /api/hosts/:id/containers/:id/logs`, `docker logs -f --timestamps`) and the container event feed (`GET /api/hosts/:id/events/stream`, via `eventWatcher`'s `Broadcaster` in `server/sse.js`) are both one-way SSE streams the frontend consumes with `EventSource`.

**Two eslint environments** (`eslint.config.js`): `server/**`, `scripts/**`, `test/**` are CommonJS/Node; `public/js/**` is an ES module/browser environment. Keep new files in the right bucket.
